# 0006 — Output GUI toolkit: egui, swappable

- **Status:** Accepted
- **Date:** 2026-07-12
- **Deciders:** Felipe Carvajal Brown

## Context

Phase 8 (GUI Output) needs a target GUI toolkit for programs generated by Visual Rust. Two candidates researched:

**Slint** — broadest platform reach of any Rust GUI toolkit (desktop + web + embedded from one codebase), stable 1.x with an explicit API-stability commitment, richer retained-mode layouts. But codegen is two-layer (a `.slint` markup file plus Rust glue/signal-slot binding code), meaningfully more complex to emit correctly from a node graph than a flat function-call model. Licensing also has real branches: the Royalty-free license excludes embedded use; embedded requires GPLv3 (open-source obligation) or a paid Commercial license — a decision surface that would leak to end users, not just us.

**egui** — immediate-mode, so a node graph maps close to 1:1 onto generated code (single `.rs` file, no separate markup layer, no signal-slot codegen). Plain MIT/Apache-2.0, no licensing branches ever, regardless of platform. No embedded support (desktop + web only), and known layout/customization limitations compared to retained-mode toolkits.

## Decision

Default output GUI toolkit is `egui`, chosen for codegen simplicity and licensing simplicity over Slint's broader platform reach. Implemented behind a swappable GUI-backend abstraction (a trait boundary between the IR and the concrete widget codegen), so an alternative backend (Slint or otherwise) can be added later without redesigning the IR — consistent with the target-agnostic IR principle in [ADR-0005](0005-target-agnostic-ir-rust-primary.md).

## Consequences

- Phase 8 embedded-target output is not available at 1.0 (egui doesn't support it); revisit if embedded output is ever prioritized, likely via adding Slint as a second backend rather than replacing egui.
- The GUI-backend trait boundary is extra up-front design work in Phase 8, not deferred work — it needs to exist before the first (`egui`) backend is implemented, not bolted on after.
- No end-user-facing licensing decision needed at 1.0, unlike a Slint-first choice would have required.

## Alternatives considered

- **Slint** — see Context above for the full comparison. Rejected as the *default* specifically because its two-layer codegen (markup + glue + signal-slot bindings) and branching embedded/GPLv3/commercial licensing are real added complexity for a v1, not because it's a worse toolkit in general. Remains the natural second backend if embedded output or richer layouts are ever prioritized.
- **gtk-rs** — mature, but Linux-first with real friction bundling GTK on Windows/macOS; ruled out early in research, not seriously contended against egui/Slint.
- **Tauri** (webview + HTML/CSS/JS) — ruled out on principle: embeds a JS/webview layer, which conflicts with the "Zero Interpreted Overhead" core principle for generated output.
